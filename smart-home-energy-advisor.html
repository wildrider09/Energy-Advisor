Project Overview

A complete end-to-end smart home energy management system built for the Amazon Smart Thermostat

🌡️

HVAC Simulation

A physics-based thermostat simulator generates realistic HVAC events every 2 seconds. It models indoor/outdoor temperature drift, compressor cycles, auxiliary heater activation, fan operation, and demand response lifecycle events. The simulator uses real weather data from Bangalore via the Open-Meteo API to drive outdoor temperature readings.

🔍

Anomaly Detection

A real-time anomaly engine evaluates device state on every simulation tick. It detects 5 categories of anomalies: auxiliary heater overuse, filter degradation, runtime spikes, demand response preparation needs, and connectivity loss. Each anomaly includes a human-readable description and actionable recommendation.

🤖

AI-Powered Chat

Claude on Amazon Bedrock powers the conversational interface. The AI receives full device context — current state, recent events, active anomalies, weather data — and responds in simple, homeowner-friendly language. It provides dollar estimates, time projections, and specific actionable recommendations.

📊

Live Dashboard

A real-time web dashboard shows indoor/outdoor temperature, HVAC component status, runtime bars, energy costs, demand response state, and Energy Star score. It polls the backend every 1.5 seconds for state updates and every 3 seconds for proactive alerts from the anomaly engine.

⚡

Demand Response

The simulator models a complete demand response lifecycle: SCHEDULED → PRE_COOLING → ACTIVE → RECOVERY → NONE. Events occur 3-4 times per simulated week. During DR events, the system automatically adjusts setpoints to reduce grid load while maintaining comfort. The AI explains what's happening and why.

🎛️

AI Optimization

Every 60 seconds, Bedrock analyzes the current state, weather forecast, temperature trends, and user preferences to decide if the thermostat settings should change. It responds with JSON containing mode, setpoint, and reason. Users can override the AI and it learns from their manual adjustments.

What Makes This Project Unique

Unlike typical demo apps, this project simulates a complete IoT device lifecycle with realistic physics, real weather data, and production-grade AI integration. Here's what sets it apart:

1

Real Physics, Not Random Data

The HVAC simulator models actual thermal dynamics — indoor temperature drifts toward outdoor temperature at a configurable rate, cooling and heating have different BTU capacities, and the auxiliary heater kicks in when the primary heater can't keep up. This creates realistic patterns that the anomaly engine can detect.

2

Live Weather Integration

The Open-Meteo API provides real-time weather data for Bangalore, India — current temperature, humidity, wind speed, feels-like temperature, and a 6-hour forecast. The weather client caches data for 10 minutes and computes temperature trends (RISING, FALLING, STABLE) that feed into AI optimization decisions.

3

Context-Rich AI Prompts

Every Bedrock call includes the full device state, last 50 events, active anomalies, weather summary, forecast, temperature trend, and learned user preferences. The system prompt instructs Claude to speak like a "helpful neighbor" — no jargon, dollar estimates, and one clear action per response.

4

Proactive Intelligence

The system doesn't wait for users to ask questions. It proactively alerts when anomalies are detected, when weather changes significantly, and when the AI auto-adjusts the thermostat. Alerts appear in the chat as advisor messages with emoji indicators.

5

User Preference Learning

When users manually override the thermostat, the system records the pattern — what mode, what temperature, at what outdoor conditions, and what time. These patterns are injected into the AI optimization prompt so Bedrock learns to respect user preferences over time.

The 3 Intelligence Pillars

Every feature in the app maps to one of three core intelligence pillars

PILLAR 1

🔧 HVAC Health Intelligence

Continuously monitors the health of your HVAC system by analyzing component behavior, runtime patterns, and maintenance indicators. The system detects problems before they become expensive failures.

Anomaly Detectors

  • Auxiliary Heater Overuse — Triggers when the aux heater runs 3+ times in a day while outdoor temperature is above 45°F. At those temperatures, the primary heater should handle the load alone. This often indicates a clogged filter, refrigerant leak, or failing compressor. The system estimates $15-20/month in wasted energy.
  • Filter Degradation — Tracks total HVAC runtime hours since the last filter change. When runtime exceeds 450 hours AND auxiliary heater usage is trending up, it alerts the homeowner. A dirty filter restricts airflow, forcing the system to work harder and use the backup heater more frequently — increasing costs 2-3x.
  • Connectivity Loss — Detects when the thermostat goes offline. Reports the last known state (mode, indoor temp, outdoor temp) so the homeowner knows what conditions the home was left in. Severity is CRITICAL because an offline thermostat can't respond to temperature changes or DR events.

How It Works in the Simulator

The EventGenerator seeds the simulation with 200 hours of total runtime and 2 aux heater cycles already recorded. As the simulation progresses, the anomaly engine evaluates state on every tick. When conditions match a detection rule, it creates an Anomaly object with a unique ID, severity level, description, and recommendation. The AdvisorController then sends the anomaly to Bedrock to generate a friendly, homeowner-readable alert.

💡 Real-World Application

On a real Snyder thermostat, these detectors would analyze actual telemetry events from the device. The physics would be real — not simulated — but the anomaly detection logic and AI integration would work identically.

PILLAR 2

💰 Energy Bill Explainability

Correlates HVAC runtime with real weather data to explain why energy costs are what they are. Instead of just showing a number, the system tells you why your bill is high and what's driving the cost.

Cost Calculation Model

  • Cooler Cost — Runtime hours × 3.5 kW (typical AC draw) × $0.16/kWh. A cooler running 5 hours costs approximately $2.80/day or $84/month.
  • Heater Cost — Runtime hours × 10.0 kW (electric heater draw) × $0.16/kWh. Electric heating is significantly more expensive than cooling.
  • Auxiliary Heater Cost — Runtime hours × 15.0 kW (resistance heating) × $0.16/kWh. Aux heating is the most expensive HVAC operation — 4x the cost of cooling per hour.
  • Monthly Projection — Today's total cost × 30 days. Simple but effective for giving homeowners a sense of their trajectory.

Anomaly Detectors

  • Runtime Spike — Triggers when the cooler has run more than 6 hours in a day while outdoor temperature is below 90°F. At moderate temperatures, 6+ hours of cooling suggests the system is working harder than it should — possibly due to poor insulation, a low setpoint, or equipment issues.

Savings Analysis

The /api/savings endpoint calculates how much the AI saves compared to running without optimization:

  • AI Optimization — 15% reduction through smart setpoint adjustments
  • Filter Maintenance — 10% reduction from clean filter efficiency
  • DR Participation — $0.40/hour × 2 hours × 3.5 events/week savings
📊 Weather Correlation

When a user asks "Why is my energy bill high?", the AI receives the full weather context — current temperature, humidity, wind, feels-like temperature, and 6-hour forecast. It can explain that a 95°F day with high humidity forces the AC to run longer, or that a cold snap triggered expensive auxiliary heating.

PILLAR 3

🌿 Demand Response & Clean Energy Advisor

Prepares for demand response events, explains PRISM impact, and suggests pre-cooling/heating strategies to maintain comfort while reducing grid load during peak demand periods.

DR Event Lifecycle

📋 SCHEDULED (1:30 PM)

A demand response event is announced. The system notifies the homeowner and begins planning. The anomaly engine creates a DR_PREPARATION anomaly with pre-cooling recommendations.

❄️ PRE_COOLING (1:45 PM)

The system lowers the setpoint by 2°F to build a thermal buffer. This "banks" cold air in the home so it stays comfortable when the AC reduces output during the event. Pre-cooling takes about 45 minutes based on the home's cooling pattern.

⚡ ACTIVE (3:00 PM)

The DR event begins. The setpoint is raised by 4°F (net +2°F from original) to reduce grid load. The Energy Star score gets a +5 bonus for DR participation. The AI explains what's happening and reassures the homeowner.

🔄 RECOVERY (5:00 PM)

The event ends. The setpoint is lowered by 2°F to begin restoring comfort. The system avoids snapping back to the original setpoint immediately to prevent a demand spike (the "rebound effect").

✅ NONE (5:30 PM)

Recovery complete. Normal operation resumes. The system calculates how much energy was saved during the event and reports it to the homeowner.

DR Event Frequency

The simulator generates DR events 3-4 times per simulated week, matching real-world patterns in regions with active demand response programs. Events occur on Monday, Wednesday, Thursday, and occasionally Friday — typically during afternoon peak hours (2-5 PM).

⚡ PRISM Integration

PRISM (Proactive Resource and Intelligent Scheduling Manager) is Amazon's clean energy optimization framework. In a production deployment, the DR advisor would integrate with PRISM to align thermostat behavior with grid carbon intensity — running the AC harder when the grid is clean (solar/wind) and reducing load when it's dirty (coal/gas).

Component Details

🌡️

EventGenerator

The heart of the simulation. Runs on a 2-second tick cycle via Spring's @Scheduled annotation. Each tick advances simulated time by 5 minutes and generates thermostat events based on HVAC physics constants:

  • Cooling rate: 0.15°F/min when cooler runs
  • Heating rate: 0.12°F/min for primary heater
  • Aux heating rate: 0.20°F/min (faster but expensive)
  • Drift rate: 0.03°F/min toward outdoor temp
💾

EventStore

Persists all thermostat events in an H2 in-memory database. Creates an indexed events table on startup with columns for device_id, event_type, event_value, numeric_value, and timestamp. Supports queries by device, type, time range, and count.

🔍

AnomalyEngine

Evaluates device state on every simulation tick. Maintains a thread-safe list of active anomalies using CopyOnWriteArrayList. Deduplicates by checking if an unacknowledged anomaly of the same category already exists for the device.

🤖

BedrockClient

Wraps the AWS SDK's BedrockRuntimeClient. Sends Messages API requests to Claude with system prompts and user messages. Uses ProfileCredentialsProvider for auth. Max tokens set to 1024 for concise responses.

📝

PromptBuilder

Constructs context-rich prompts for different scenarios: chat responses, anomaly alerts, and thermostat optimization. Injects full device state, recent events, anomalies, weather data, and user preferences into every prompt.

☀️

WeatherClient

Fetches real weather data from Open-Meteo API for Bangalore (12.97°N, 77.59°E). Caches for 10 minutes. Provides current conditions, 6-hour forecast, and temperature trend (RISING/FALLING/STABLE) based on 3-hour lookahead.

Scheduled Tasks

Task Interval Initial Delay Description
simulationTick() 2 seconds None Advances HVAC simulation, saves events, evaluates anomalies
autoOptimize() 60 seconds 15 seconds Bedrock analyzes state and adjusts thermostat settings
weatherFeedback() 5 minutes 30 seconds Proactive weather-based tips when conditions change

Concurrency Model

The application uses Spring's default thread pool for scheduled tasks. Thread safety is achieved through:

  • CopyOnWriteArrayList for active anomalies and proactive alerts
  • volatile boolean for manual override flag in EventGenerator
  • Atomic state updates within the single-threaded simulation tick
  • H2's built-in connection pooling for database access

5.2 BedrockClient — AI Integration

Wraps the AWS Bedrock SDK to send chat requests to Claude. Uses the Messages API format with system prompts for context injection.

src/main/java/com/amazon/energyadvisor/ai/BedrockClient.java Java 
package com.amazon.energyadvisor.ai;

import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.stereotype.Component;
import software.amazon.awssdk.auth.credentials.ProfileCredentialsProvider;
import software.amazon.awssdk.core.SdkBytes;
import software.amazon.awssdk.regions.Region;
import software.amazon.awssdk.services.bedrockruntime.BedrockRuntimeClient;
import software.amazon.awssdk.services.bedrockruntime.model.InvokeModelRequest;
import software.amazon.awssdk.services.bedrockruntime.model.InvokeModelResponse;
import com.fasterxml.jackson.databind.JsonNode;
import com.fasterxml.jackson.databind.ObjectMapper;
import com.fasterxml.jackson.databind.node.ArrayNode;
import com.fasterxml.jackson.databind.node.ObjectNode;
import jakarta.annotation.PostConstruct;

@Component
public class BedrockClient {

    private static final Logger log = LoggerFactory.getLogger(BedrockClient.class);
    private final ObjectMapper mapper = new ObjectMapper();

    @Value("${bedrock.model-id}")
    private String modelId;

    @Value("${bedrock.region:us-east-1}")
    private String region;

    private BedrockRuntimeClient client;

    @PostConstruct
    public void init() {
        client = BedrockRuntimeClient.builder()
                .region(Region.of(region))
                .credentialsProvider(ProfileCredentialsProvider.create())
                .build();
        log.info("Bedrock client initialized: model={}, region={}", modelId, region);
    }

    public String chat(String systemPrompt, String userMessage) {
        ObjectNode body = mapper.createObjectNode();
        body.put("anthropic_version", "bedrock-2023-05-31");
        body.put("max_tokens", 1024);
        body.put("system", systemPrompt);

        ArrayNode messages = body.putArray("messages");
        ObjectNode msg = messages.addObject();
        msg.put("role", "user");
        ArrayNode content = msg.putArray("content");
        content.addObject().put("type", "text").put("text", userMessage);

        try {
            InvokeModelRequest request = InvokeModelRequest.builder()
                    .modelId(modelId)
                    .contentType("application/json")
                    .accept("application/json")
                    .body(SdkBytes.fromUtf8String(mapper.writeValueAsString(body)))
                    .build();

            InvokeModelResponse response = client.invokeModel(request);
            JsonNode result = mapper.readTree(response.body().asUtf8String());
            return result.path("content").get(0).path("text").asText();
        } catch (Exception e) {
            log.error("Bedrock call failed: {}", e.getMessage());
            throw new RuntimeException("Bedrock invocation failed", e);
        }
    }
}
🔑 Key Design Decisions

ProfileCredentialsProvider — Uses ~/.aws/credentials for auth. In production, you'd use IAM roles or container credentials.
max_tokens: 1024 — Keeps responses concise. The system prompt asks for 3-5 sentences, so 1024 tokens is plenty.
anthropic_version — Required by Bedrock's Claude integration. Must match the API version.

5.3 PromptBuilder — Context-Rich Prompts

Constructs detailed system prompts that give Claude full context about the thermostat state, recent events, anomalies, and weather. This is what makes the AI responses specific and actionable rather than generic.

src/main/java/com/amazon/energyadvisor/ai/PromptBuilder.java Java 
package com.amazon.energyadvisor.ai;

import com.amazon.energyadvisor.model.Anomaly;
import com.amazon.energyadvisor.model.DeviceState;
import org.springframework.stereotype.Component;
import java.util.List;
import java.util.Map;

@Component
public class PromptBuilder {

    public String buildSystemPrompt(DeviceState state,
            List<Map<String, Object>> recentEvents,
            List<Anomaly> anomalies) {

        StringBuilder sb = new StringBuilder();

        // Persona instructions — speak like a helpful neighbor
        sb.append("You are a friendly Smart Home Energy Advisor. ");
        sb.append("You talk to regular homeowners — NOT engineers.\n");
        sb.append("- Use simple, everyday language.\n");
        sb.append("- Say 'your AC' not 'cooler', 'backup heater' not 'auxiliary heater'.\n");
        sb.append("- Keep answers to 3-5 sentences.\n");
        sb.append("- Always include a specific dollar amount or time estimate.\n");
        sb.append("- End with ONE clear action the customer can take right now.\n\n");

        // Full device state injection
        sb.append("CURRENT DEVICE STATE:\n");
        sb.append(String.format("- Mode: %s\n", state.getThermostatMode()));
        sb.append(String.format("- Target Setpoint: %.1f°F\n", state.getTargetSetpoint()));
        sb.append(String.format("- Indoor Temp: %.1f°F\n", state.getIndoorTemp()));
        sb.append(String.format("- Outdoor Temp: %.1f°F\n", state.getOutdoorTemp()));
        // ... heater, cooler, DR status, runtime hours ...

        // Active anomalies
        if (!anomalies.isEmpty()) {
            sb.append("\nACTIVE ANOMALIES:\n");
            for (Anomaly a : anomalies) {
                sb.append(String.format("- [%s] %s: %s\n",
                    a.getSeverity(), a.getCategory(), a.getDescription()));
            }
        }

        // Recent event timeline (last 50 events)
        if (!recentEvents.isEmpty()) {
            sb.append("\nRECENT EVENT TIMELINE:\n");
            for (Map<String, Object> e : recentEvents) {
                sb.append(String.format("- %s: %s = %s\n",
                    e.get("TS"), e.get("EVENT_TYPE"), e.get("EVENT_VALUE")));
            }
        }

        return sb.toString();
    }

    // Anomaly alert prompt — 2 sentences max, emoji, dollar estimate
    public String buildAnomalyPrompt(Anomaly anomaly, DeviceState state) {
        return String.format(
            "Write a friendly alert for a homeowner. " +
            "Issue: %s — %s. Indoor: %.1f°F, Outdoor: %.1f°F. " +
            "Recommendation: %s. Use simple language. 2 sentences max.",
            anomaly.getCategory(), anomaly.getDescription(),
            state.getIndoorTemp(), state.getOutdoorTemp(),
            anomaly.getRecommendation());
    }

    // Optimization prompt — returns JSON with mode, setpoint, reason
    public String buildOptimizePrompt(DeviceState state,
            String weatherSummary, List<Anomaly> anomalies,
            String forecast, String trend) {

        StringBuilder sb = new StringBuilder();
        sb.append("You are an AI thermostat optimizer.\n");
        sb.append(String.format("STATE: mode=%s, setpoint=%.1f°F, " +
            "indoor=%.1f°F, outdoor=%.1f°F\n",
            state.getThermostatMode(), state.getTargetSetpoint(),
            state.getIndoorTemp(), state.getOutdoorTemp()));
        sb.append(String.format("Weather: %s\nForecast: %s\nTrend: %s\n",
            weatherSummary, forecast, trend));

        // Optimization rules
        sb.append("Rules:\n");
        sb.append("- During DR: raise setpoint 2-4°F\n");
        sb.append("- If outdoor < 75°F and comfortable: switch OFF\n");
        sb.append("- If outdoor > 95°F: ensure COOL, don't go below 72°F\n");
        sb.append("- If trend FALLING: pre-adjust to avoid overcooling\n");
        sb.append("- If trend RISING: pre-cool before it gets hotter\n\n");
        sb.append("Respond ONLY with JSON: {\"mode\":\"COOL|HEAT|OFF\", " +
            "\"setpoint\": <number>, \"reason\": \"...\"}\n");

        return sb.toString();
    }
}

7.1 EventGenerator — HVAC Physics Simulator

The most complex class in the project. Simulates realistic HVAC behavior including thermal drift, compressor cycling, auxiliary heater activation, demand response lifecycle, and Energy Star scoring. Each tick represents 5 simulated minutes.

🌡️ Physics Model

Indoor temperature drifts toward outdoor temperature at 0.03°F/min (natural heat transfer). When the cooler runs, it pulls temperature down at 0.15°F/min. The primary heater pushes up at 0.12°F/min, and the aux heater at 0.20°F/min. These rates create realistic oscillation patterns around the setpoint.

src/main/java/com/amazon/energyadvisor/simulator/EventGenerator.java Java 
@Component
public class EventGenerator {

    private static final String DEVICE_ID = "snyder-demo-001";

    // HVAC physics constants
    private static final double COOLING_RATE = 0.15;    // °F/min when cooler runs
    private static final double HEATING_RATE = 0.12;    // °F/min primary heater
    private static final double AUX_HEATING_RATE = 0.20; // °F/min aux heater
    private static final double DRIFT_RATE = 0.03;      // °F/min toward outdoor

    private final WeatherClient weather;
    private final DeviceState state = new DeviceState(DEVICE_ID);
    private Instant simTime = Instant.now();
    private int tickCount = 0;
    private volatile boolean manualOverride = false;

    public EventGenerator(WeatherClient weather) {
        this.weather = weather;
        // Seed realistic state — Snyder has been running all day
        state.setThermostatMode("COOL");
        state.setTargetSetpoint(74);
        state.setIndoorTemp(75.2);
        state.setOutdoorTemp(weather.getOutdoorTemp());
        state.setCoolerRunning(true);
        state.setCoolerRuntimeToday(9000);         // 2.5h cooling today
        state.setAuxHeaterCyclesToday(2);           // aux kicked in twice
        state.setTotalHvacRuntimeHours(200);        // filter getting old
        tickCount = 453;                            // near a DR event
    }

    /** Advance simulation by one tick (~5 simulated minutes) */
    public List<ThermostatEvent> tick() {
        List<ThermostatEvent> events = new ArrayList<>();
        tickCount++;
        simTime = simTime.plusSeconds(300); // 5 min per tick

        // Use real outdoor temp from Bangalore weather
        state.setOutdoorTemp(weather.getOutdoorTemp());

        // HVAC physics: drift indoor temp toward outdoor
        double tempDelta = state.getOutdoorTemp() - state.getIndoorTemp();
        state.setIndoorTemp(state.getIndoorTemp() + tempDelta * DRIFT_RATE);

        // Thermostat control logic
        String mode = state.getThermostatMode();
        double target = state.getTargetSetpoint();

        if ("COOL".equals(mode)) {
            // Start cooler, accumulate runtime, pull temp down
            state.setCoolerRuntimeToday(state.getCoolerRuntimeToday() + 2);
            if (state.getIndoorTemp() > target) {
                state.setIndoorTemp(state.getIndoorTemp() - COOLING_RATE * 5);
            }
        } else if ("HEAT".equals(mode)) {
            // Primary heater with aux heater fallback
            if (state.getIndoorTemp() < target - 1.0) {
                double rate = HEATING_RATE;
                // Aux heater kicks in when outdoor < 40°F
                if (state.getOutdoorTemp() < 40) {
                    state.setAuxHeaterRunning(true);
                    state.setAuxHeaterCyclesToday(
                        state.getAuxHeaterCyclesToday() + 1);
                    rate = AUX_HEATING_RATE;
                }
                state.setIndoorTemp(state.getIndoorTemp() + rate * 5);
            }
        }

        // Always emit temperature reading
        events.add(tempEvent());

        // DR events: 3-4 per week, 5-state lifecycle
        int dayTick = tickCount % 288;  // 288 ticks = 24h
        int weekDay = (tickCount / 288) % 7;

        if (drDay && dayTick == 162) {  // 1:30 PM: schedule
            state.setDrStatus("SCHEDULED");
        } else if (drDay && dayTick == 171) {  // 1:45 PM: pre-cool
            state.setDrStatus("PRE_COOLING");
            state.setTargetSetpoint(target - 2);
        } else if (drDay && dayTick == 180) {  // 3 PM: active
            state.setDrStatus("ACTIVE");
            state.setTargetSetpoint(target + 4);
        } else if (drDay && dayTick == 204) {  // 5 PM: recovery
            state.setDrStatus("RECOVERY");
        } else if (drDay && dayTick == 210) {  // 5:30 PM: done
            state.setDrStatus("NONE");
        }

        // Energy Star V2 score calculation
        int score = 85;
        if (coolerH > 8) score -= 10;        // excessive cooling
        if (auxH > 0.5) score -= 8;          // aux heater overuse
        if (filterH > 300) score -= 5;       // dirty filter
        if (drActive) score += 5;            // DR participation bonus
        if (!manualOverride) score += 3;    // AI optimization bonus
        state.setEnergyStarScore(Math.max(0, Math.min(100, score)));

        return events;
    }
}

7.2 AnomalyEngine — Real-Time Detection

Evaluates device state on every tick and creates anomaly objects when conditions match. Uses deduplication to avoid flooding the user with repeated alerts.

src/main/java/com/amazon/energyadvisor/engine/AnomalyEngine.java Java 
@Component
public class AnomalyEngine {

    private final List<Anomaly> activeAnomalies = new CopyOnWriteArrayList<>();

    /** Evaluate device state and return any new anomalies */
    public List<Anomaly> evaluate(DeviceState state) {
        List<Anomaly> newAnomalies = new ArrayList<>();

        // Pillar 1: Aux heater overuse
        if (state.getAuxHeaterCyclesToday() >= 3
                && state.getOutdoorTemp() > 45) {
            if (noActiveAnomaly(state.getDeviceId(),
                    Category.AUX_HEATER_OVERUSE)) {
                Anomaly a = new Anomaly(
                    state.getDeviceId(),
                    Category.AUX_HEATER_OVERUSE,
                    Severity.WARNING,
                    "Auxiliary heater has run 3+ times at mild temps",
                    "Replace air filter — could save $15-20/month");
                activeAnomalies.add(a);
                newAnomalies.add(a);
            }
        }

        // Pillar 1: Filter degradation
        if (state.getTotalHvacRuntimeHours() > 450
                && state.getAuxHeaterCyclesToday() >= 2) {
            if (noActiveAnomaly(state.getDeviceId(),
                    Category.FILTER_DEGRADATION)) {
                // Alert: filter needs replacement
            }
        }

        // Pillar 2: Runtime spike
        if (state.getCoolerRuntimeToday() > 6 * 3600
                && state.getOutdoorTemp() < 90) {
            // Alert: cooler running excessively
        }

        // Pillar 3: DR preparation
        if ("SCHEDULED".equals(state.getDrStatus())) {
            // Alert: pre-cool recommendation
        }

        // Connectivity loss
        if (!state.isOnline()) {
            // CRITICAL alert: device offline
        }

        return newAnomalies;
    }

    // Deduplication: only one active anomaly per category per device
    private boolean noActiveAnomaly(String deviceId, Category cat) {
        return activeAnomalies.stream()
            .noneMatch(a -> a.getDeviceId().equals(deviceId)
                && a.getCategory() == cat
                && !a.isAcknowledged());
    }
}

7.3 EventStore — H2 Persistence

Persists thermostat events in an H2 in-memory database. Creates the schema on startup and provides query methods for recent events, events by type, and event counts.

src/main/java/com/amazon/energyadvisor/store/EventStore.java Java 
@Repository
public class EventStore {

    private final JdbcTemplate jdbc;

    @PostConstruct
    public void init() {
        jdbc.execute("""
            CREATE TABLE IF NOT EXISTS events (
                id BIGINT AUTO_INCREMENT PRIMARY KEY,
                device_id VARCHAR(64),
                event_type VARCHAR(32),
                event_value VARCHAR(256),
                numeric_value DOUBLE,
                ts TIMESTAMP
            )
        """);
        jdbc.execute("CREATE INDEX IF NOT EXISTS idx_device_ts "
            + "ON events(device_id, ts)");
    }

    public void save(ThermostatEvent event) {
        jdbc.update(
            "INSERT INTO events "
            + "(device_id, event_type, event_value, numeric_value, ts) "
            + "VALUES (?,?,?,?,?)",
            event.getDeviceId(),
            event.getEventType().name(),
            event.getValue(),
            event.getNumericValue(),
            Timestamp.from(event.getTimestamp()));
    }

    public void saveAll(List<ThermostatEvent> events) {
        events.forEach(this::save);
    }

    public List<Map<String, Object>> getRecentEvents(
            String deviceId, int limit) {
        return jdbc.queryForList(
            "SELECT event_type, event_value, numeric_value, ts "
            + "FROM events WHERE device_id = ? "
            + "ORDER BY ts DESC LIMIT ?",
            deviceId, limit);
    }
}

7.4 WeatherClient — Live Weather Data

Fetches real weather data from the Open-Meteo API for Bangalore, India. Caches results for 10 minutes. Provides current conditions, 6-hour forecast, and temperature trend.

src/main/java/com/amazon/energyadvisor/weather/WeatherClient.java Java 
@Component
public class WeatherClient {

    private final HttpClient http = HttpClient.newHttpClient();
    private final ObjectMapper mapper = new ObjectMapper();

    @Value("${weather.default-lat:12.9716}")
    private String lat;

    @Value("${weather.default-lon:77.5946}")
    private String lon;

    private double cachedTempF = Double.NaN;
    private String cachedSummary = "";
    private String cachedForecast = "";
    private String cachedTrend = "STABLE";
    private long cachedAt = 0;

    private void refreshIfStale() {
        if (System.currentTimeMillis() - cachedAt < 600_000) return;
        try {
            String url = String.format(
                "https://api.open-meteo.com/v1/forecast?"
                + "latitude=%s&longitude=%s"
                + "&current=temperature_2m,relative_humidity_2m,"
                + "weather_code,wind_speed_10m,apparent_temperature"
                + "&hourly=temperature_2m,weather_code"
                + "&forecast_hours=6"
                + "&temperature_unit=fahrenheit", lat, lon);

            HttpResponse<String> resp = http.send(
                HttpRequest.newBuilder().uri(URI.create(url)).GET().build(),
                HttpResponse.BodyHandlers.ofString());

            JsonNode root = mapper.readTree(resp.body());
            JsonNode current = root.path("current");

            cachedTempF = current.path("temperature_2m").asDouble(85.0);
            double humidity = current.path("relative_humidity_2m").asDouble();
            double feelsLike = current.path("apparent_temperature").asDouble();

            cachedSummary = String.format(
                "%.0f°F (feels like %.0f°F), humidity %.0f%%",
                cachedTempF, feelsLike, humidity);

            // Compute trend from 3-hour lookahead
            JsonNode hourlyTemps = root.path("hourly")
                .path("temperature_2m");
            if (hourlyTemps.size() >= 3) {
                double later = hourlyTemps.get(2).asDouble();
                if (later > cachedTempF + 3) cachedTrend = "RISING";
                else if (later < cachedTempF - 3) cachedTrend = "FALLING";
                else cachedTrend = "STABLE";
            }

            cachedAt = System.currentTimeMillis();
        } catch (Exception e) {
            log.warn("Open-Meteo API failed: {}", e.getMessage());
        }
    }
}

Energy Star Score & Cost Model

How the app calculates efficiency scores and energy costs

Energy Star V2 Score Calculation

The Energy Star score is a 0-100 rating calculated on every simulation tick. It starts at a baseline of 85 and adjusts based on runtime efficiency:

FactorImpactCondition
Baseline85Starting score
Excessive cooling-10Cooler runtime > 8 hours today
Aux heater overuse-8Aux runtime > 0.5 hours today
Dirty filter-5Total runtime > 300 hours
DR participation+5DR status is ACTIVE or PRE_COOLING
AI optimization+3Manual override is OFF (AI controlling)

Cost Calculation Details

Energy costs are calculated using typical US residential electricity rates and standard HVAC power draws:

ComponentPower DrawRateCost/HourExample (5h)
❄️ AC Compressor 3.5 kW $0.16/kWh $0.56/hr $2.80
🔥 Primary Heater 10.0 kW $0.16/kWh $1.60/hr $8.00
⚡ Aux Heater 15.0 kW $0.16/kWh $2.40/hr $12.00
⚡ Why Aux Heating is So Expensive

Auxiliary heating uses resistance coils — essentially a giant toaster inside your HVAC. At 15 kW, it draws 4.3x more power than the AC compressor. Even 30 minutes of aux heating costs $1.20 — the same as running the AC for over 2 hours. This is why the anomaly engine flags aux heater overuse as a WARNING.

Savings Model

🤖

AI Optimization: 15%

Smart setpoint adjustments based on weather, time of day, and user preferences reduce unnecessary runtime by ~15%.

🔧

Filter Maintenance: 10%

A clean filter improves airflow efficiency, reducing the load on the compressor and eliminating unnecessary aux heater activation.

⚡

DR Participation: ~$12/mo

Reducing load during 3.5 DR events/week × 2 hours × $0.40/hour savings = approximately $12/month in avoided peak pricing.

Design Decisions

Why things are built the way they are

Why In-Memory H2 Instead of a Real Database?

This is a demo application. H2 in-memory mode means zero setup — no Docker, no database server, no connection strings. The data resets on restart, which is fine for a simulator. In production, you'd swap to DynamoDB or PostgreSQL by changing the datasource URL in application.yml.

Why Vanilla JS Instead of React/Vue?

The frontend is ~250 lines of JavaScript. Adding React would require a build step (webpack/vite), node_modules, and JSX compilation — all for a dashboard that polls 4 endpoints and updates DOM elements. Vanilla JS keeps the project self-contained: ./gradlew bootRun and you're done.

Why Polling Instead of WebSockets?

WebSockets would be more efficient for real-time updates, but polling at 1.5-second intervals is simpler to implement and debug. The overhead is minimal — each poll is a small JSON response. For a demo with one user, polling is the right trade-off.

Why ProfileCredentialsProvider for Bedrock?

For a local demo, reading from ~/.aws/credentials is the simplest auth method. In production on EC2/ECS/Lambda, you'd use DefaultCredentialsProvider which automatically picks up IAM role credentials.

Why 5 Simulated Minutes Per 2-Second Tick?

At 1:1 time, you'd wait hours to see a DR event or filter degradation. The 150x speedup (5 min / 2 sec) means a full simulated day passes in ~9.6 real minutes. You see DR events, mode transitions, and anomalies within the first few minutes of running the app.

Why Bangalore Weather?

Bangalore has interesting weather for HVAC simulation — warm enough to need cooling most of the year, but with enough variation (monsoon season, cool mornings) to trigger different HVAC modes. The Open-Meteo API provides free, no-key-required weather data for any location.

Glossary

Key terms used throughout this documentation

TermDefinition
HVACHeating, Ventilation, and Air Conditioning — the system that controls indoor climate.
SetpointThe target temperature the thermostat tries to maintain.
Auxiliary HeaterA backup resistance heater that activates when the primary heater can't keep up. Very expensive to run.
Demand Response (DR)A program where utilities ask customers to reduce electricity usage during peak demand periods.
Pre-coolingLowering the temperature before a DR event to build a thermal buffer.
PRISMProactive Resource and Intelligent Scheduling Manager — Amazon's clean energy optimization framework.
Energy StarA US government program that rates energy efficiency. V2 is the latest thermostat standard.
SnyderAmazon's smart thermostat product codename.
BedrockAmazon's managed AI service that provides access to foundation models like Claude.
ClaudeAnthropic's AI model, accessed through Amazon Bedrock for conversational responses.
Open-MeteoA free weather API that provides current conditions and forecasts without requiring an API key.
H2A lightweight Java SQL database that can run in-memory. Zero configuration required.
Thermal DriftThe natural tendency of indoor temperature to move toward outdoor temperature over time.
Rebound EffectA demand spike that occurs when many homes simultaneously restore their setpoints after a DR event.
